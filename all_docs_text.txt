Problem Statement Document
Document 01
Prepared for pre-coding planning, implementation readiness, and project evaluation.

Purpose
This document defines the exact problem AIRWATCH solves and frames it for technical development, hackathon judging, and real rural deployment.
Exact Problem
Large rural populations in India are exposed to poor air quality caused by biomass burning, industrial pollution, crop residue burning, household smoke, and waste burning. Most villages do not have affordable monitoring infrastructure, public warning systems, or localized awareness tools.
Root Causes
High cost of official monitoring stations
Low internet availability in rural regions
Limited awareness of AQI and health risks
No village-level historic pollution data
Language barriers in public warnings
Lack of community incentives to reduce pollution
Expected Outcomes
Enable low-cost air quality monitoring
Provide real-time environmental insights
Generate local alerts and awareness
Support data-driven decisions by panchayats and administrators
Work effectively in low-connectivity regions
Target Users

Success Statement
AIRWATCH succeeds when a village can detect harmful pollution locally, alert residents without internet dependency, and use quarterly rankings to encourage cleaner practices.

[TABLE 1]
Stakeholder | Need
Villagers | Understand air safety in local language
Panchayat/Admin | Monitor village-wide pollution
Schools | Protect children during high AQI events
Farmers | Awareness during crop residue burning periods
Government bodies | Aggregated village-level data and rankings

Project Proposal Document
Document 02

Executive Summary
AIRWATCH is a low-cost IoT and AI platform for rural air quality monitoring. It combines ESP32 sensor nodes, LoRa wireless communication, Raspberry Pi dashboard, GSM alerts, Sarvam AI multilingual support, and village-level clean-air competition.
Project Objectives
Build distributed sensor nodes for air-quality monitoring
Transmit node readings through LoRa without internet
Run a Raspberry Pi hub for dashboard, storage, prediction, and alerts
Register users with mobile number and preferred language
Send SMS and voice alerts in regional language
Provide admin map-based node installation
Rank villages quarterly for clean-air rewards
Proposed Modules

Innovation
Offline-first rural design
Local-language voice alerts
Governance dashboard instead of only sensor display
Gamified village ranking for behavior change
Map-based node installation and heatmaps
Scope
The prototype focuses on MQ135-based air-quality indication, LoRa node communication, dashboard visualization, emergency alerts, multilingual interface, and admin operations. PM2.5 sensors and solar power are future upgrades.

[TABLE 1]
Module | Description
Sensor Node | MQ135, LCD, LED, buzzer, LoRa on ESP32
Central Hub | Raspberry Pi receiving LoRa packets and hosting dashboard
Alert System | GSM SMS plus voice alert layer
Multilingual Layer | Sarvam AI translation and voice support
Admin Platform | Node map, users, analytics, broadcasts
Ranking System | Quarterly village AQI competition and rewards

Database Overview
SQLite is used for the prototype because it is lightweight and works offline on Raspberry Pi. It can later be migrated to PostgreSQL or cloud database.

Relationships
One village has many users.
One village has many nodes.
One node has many readings.
Alerts may belong to a node and optionally to a user.
Rankings are calculated per village per quarter.
Sample SQL Schema
CREATE TABLE users (id INTEGER PRIMARY KEY AUTOINCREMENT, name TEXT, email TEXT UNIQUE, password_hash TEXT, mobile TEXT, village_id INTEGER, preferred_language TEXT, role TEXT);
CREATE TABLE nodes (id INTEGER PRIMARY KEY AUTOINCREMENT, node_code TEXT UNIQUE, village_id INTEGER, location_name TEXT, latitude REAL, longitude REAL, status TEXT);
CREATE TABLE readings (id INTEGER PRIMARY KEY AUTOINCREMENT, node_id INTEGER, mq135_value INTEGER, aqi_value INTEGER, category TEXT, battery INTEGER, timestamp DATETIME DEFAULT CURRENT_TIMESTAMP);

[TABLE 1]
Table | Fields
users | id, name, email, password_hash, mobile, village_id, preferred_language, role, created_at
villages | id, name, district, state, default_language
nodes | id, node_code, village_id, location_name, latitude, longitude, status, installed_at
readings | id, node_id, mq135_value, aqi_value, category, battery, timestamp
alerts | id, node_id, user_id, alert_type, language, message, delivery_status, timestamp
rankings | id, village_id, quarter, avg_aqi, improvement_percent, rank, reward_status

AIRWATCH
Rural Environmental Intelligence and Air Quality Governance Platform

LoRa Communication Protocol
Document 10
Prepared for pre-coding planning, implementation readiness, and project evaluation.

Purpose
LoRa enables long-range, low-power, low-cost communication from ESP32 nodes to Raspberry Pi hub without internet or SIM cards.
Packet Format
NODE_ID|MQ135|AQI|CATEGORY|BATTERY|TIMESTAMP
Example: NODE_01|720|286|Poor|78|10:42
Transmission Rules
Send packet every 30 to 60 seconds during demo.
Use shorter interval only for live demonstration.
Include node ID in every packet.
Retry once if ACK is not received, if ACK feature is implemented.
Use checks to ignore corrupted or incomplete packets.
Receiver Behavior
Listen continuously
Decode packet
Validate node ID
Store reading in database
Update node last_seen time
Trigger alert if threshold crossed
Frequency Note
Use the frequency supported by the available LoRa module. Keep transmitter and receiver frequency, bandwidth, spreading factor, and sync word identical.

else: risk = "Low"

Risk level
Recommended action
Confidence label such as Low/Medium/High

Technical Feasibility

Operational Feasibility
Local LCD, LEDs, and buzzer require no smartphone

Risks

Conclusion
